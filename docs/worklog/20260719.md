# 19 July 2026

## Backups, and a look at how pages get created

**Set up the offline backups.** The laptop now pulls both repositories from
the git server once a day and writes dated snapshot files. Two repositories
matter now: the site's content, and the smaller one holding the scripts that
build and publish it — losing the second would mean rebuilding all of
yesterday's plumbing.

The snapshots go on the laptop's second disk, separate from the one holding
the working copies, so one drive failing does not take both. Old snapshots
are trimmed to the last twelve.

Two deliberate choices worth remembering:

* The laptop **pulls** from the server, never the other way round. The server
  has no way to reach this machine, so if it were ever compromised it could
  not reach the backups either.
* Deleted branches upstream are **not** deleted here. If something is wiped
  or overwritten on the server by mistake, the old copy survives locally.

**Tested a restore rather than assuming one.** Rebuilt the whole site from a
single snapshot file — 114 pages, no errors. A backup nobody has restored is
only a guess.

**Where this sits.** Three layers now: the git server at home is the real
copy, GitHub is the automatic off-site copy, and these snapshots add
point-in-time history that survives a mistake being copied to GitHub. The
snapshots are on the same laptop though, so they are not an off-site copy —
GitHub is the one that covers losing the machine.

**Learned where new pages actually get created.** The site itself has no
place to add a page — the edit button on a page takes you to the git server
for *that* file. To create a new one, go to the git server directly and use
**Add File → New File**, typing the full path. The page appears on the site a
few seconds later, and the menu updates on its own.

## Decided what *not* to back up

Started backing up the git server's own settings — a single file holding
accounts, keys and webhook configuration. Then thought better of it: that file
also contains passwords and the GitHub token in plain text, and those never
change, so copying them nightly onto an unencrypted disk added risk for no
benefit.

Instead the backup now copies the **password store**, which is encrypted by
design and is where those values belong. Everything else in the settings file
— one account, two repositories, a key, a webhook — is quicker to recreate
from the notes than to restore.

Also fixed the webhook receiver, which had been quietly dying whenever I
logged out of the desktop machine. Publishing now works with nobody logged in.

Wrote a separate note on [renewing an expiring GPG
key](../content/linux-admin/gpg_key_renewal.md), since the encryption above
now depends on those keys and one of them expires in 2029.

## Later: the site went public

**<https://til.anandas.in> is live.** The view-only copy now sits on the public
server, with a proper certificate. That was the last big piece.

**Publishing is deliberately manual.** It would have been easy to make the
public site follow every save, but that is not what I want — half-finished
edits would go straight out. Instead everything lands on the home copy
immediately, and one command on the desktop machine pushes the public site
forward when I decide it has caught up enough. Editing stays fast; publishing
stays a choice.

**Simplified the deploy after over-engineering it.** I first built the push
with dated release folders and a swap step so an update would be instant and
reversible. That is machinery for a site where a moment of inconsistency
matters, and this is not one. Now it simply copies the files across. The one
safeguard I kept is a check that the built site looks sane before anything is
sent — a copy command pointed at an empty folder will cheerfully erase the
site at the other end, which is close to how I wiped a set of keys by accident
last week.

## A certificate problem that was not ours

Partway through, certificate renewals started failing on the same server —
three other sites, unrelated to this one. Easy to assume we had broken
something.

We had not. Those three were set up to renew using a method that needs the web
server switched off, on a machine where the web server never switches off.
They had almost certainly been failing quietly since the day they were issued,
and only surfaced now because the certificates finally got close to expiring.
Switched all three to renew through the running web server instead, and
confirmed with a dry run that every certificate on the machine now renews
cleanly.

Worth remembering as a pattern: a failure that appears right after a change is
not necessarily caused by it, and the quickest way to tell is to ask whether
the old setup could ever have worked.

## Tidied up which machine does what

Went to change a build script and found it could only be edited on the desktop
machine — which is the machine that deploys, so it should not be a place where
work is written. Moved that editing to the laptop.

Doing so turned up something worse. The desktop machine had **no read access to
the repository holding its own build scripts**. It could never have collected
an update to them. That is why the change I had just made was stuck there with
nowhere to go: it was the only place it could live.

Fixed by giving it a read-only key for that repository too. Now every machine
has one clear job:

* **Laptop** — where things get written, and the only machine that can push
* **Desktop** — pulls, builds, and publishes; cannot write anything back
* **Public server** — serves files and nothing else

Spent a while assuming a key had been changed or revoked, when the real answer
was that the access had never existed. Worth remembering: when something has
"stopped working", check first whether it ever actually worked.

## Pinned the site's builder to its current major version

The documentation tool warned that its next major version is a rewrite that
drops the entire plugin system — every add-on this site relies on would stop
working, with no upgrade path.

Nothing to worry about today, and the theme already guards against it. But the
dependency list is deliberately unpinned, so that protection was inherited
rather than stated. Written down explicitly now, so it cannot quietly vanish
one day and take the site's build with it.

## Made the work log write itself

This diary only gets written when I remember to ask for it, which means it will
eventually stop being written. Set up a reminder that fires automatically at
the end of every session, on any project on this machine — not just this one.

The important detail is *where* the reminder lives. Asking nicely in a notes
file is advisory and can be passed over; this is enforced by the tool itself,
so it cannot be quietly skipped.

Deliberately built so it can never become a nuisance:

* It interrupts **once** per session and never again, so there is no way to get
  stuck unable to finish.
* It stays silent on machines that do not have this diary.
* It is told not to invent an entry when nothing meaningful happened.
* It may push to the home git server, which updates the intranet copy, but it
  is explicitly forbidden from publishing to the public site. That stays a
  decision I make.

The last point is the one worth keeping. The whole reason publishing is manual
is so half-finished work does not go out; an automatic reminder that quietly
published would have undone that.
