# 18 July 2026

## Republished the site, and moved it onto my own git server

Spent the day giving this site a proper home.

**Cleaned up the content first.** The site was only publishing about half of
what was in the repository — the navigation was a hand-written list, so
anything not added to it by hand simply never appeared. Fixed that: the menu
is now built from the folder structure, so a new page shows up on its own.
That took the site from 61 to 112 published pages. Unfinished notes are no
longer hidden; they are published with a *work in progress* label and
collected under [Tags](../tags.md).

**Found and removed a nasty surprise.** Every page was loading a script from
`polyfill.io`. That service changed hands a couple of years ago and started
serving malware, which is why the browser began asking for a sign-in. It had
been there for years, inherited from an old example in the MkDocs
documentation. Removed.

**Set up Forgejo on the desktop machine.** This is my own git server, running
on the intranet. The site's repository now lives there as `anandas/til`, and
GitHub has become a mirror — it updates automatically whenever I push, purely
as an off-site backup. Nothing should be edited on GitHub any more.

**The site is now built and served from home.** Two copies run on the
intranet:

| Address | What it is |
| --- | --- |
| <http://192.168.0.104:8090/> | the **editable** copy, with an *edit this page* button on every page |
| <http://192.168.0.104:8091/> | the **view-only** copy — this is what will go on the internet |

The split is deliberate. The public copy is built without any link back to the
git server, so visitors get no edit button and no clue where the repository
lives. Editing only exists on the home network.

**Publishing is now automatic.** Editing a page in the browser, saving it, and
watching it appear on the site a few seconds later works — no terminal
involved. Behind the scenes a push tells the build to run, and it refuses to
publish if anything looks wrong.

**Wrote it all down** in
[Self-hosting this TIL site](../content/linux-infra/til_website_setup.md),
including a long list of things that went wrong along the way, so the next
setup is less painful.
